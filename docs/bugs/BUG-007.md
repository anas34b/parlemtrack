# BUG-007 — port 6379 occupé par un service Redis système tiers

- Symptôme : l'API démarrait sans erreur et répondait correctement, mais aucune clé n'apparaissait jamais dans le cache Redis du conteneur Docker du projet (`docker compose exec redis redis-cli KEYS "*"` restait toujours vide), alors que le code de mise en cache (`backend/app/core/cache.py`) s'exécutait sans exception.
- Cause racine : un service Redis système, sans rapport avec le projet, tournait déjà sur le port 6379 de la machine hôte (visible via `ps aux | grep redis`, `ss -tlnp`). La connexion applicative `redis://localhost:6379/0` atteignait donc ce Redis système au lieu du conteneur Docker `parlemtrack_redis`, qui avait pourtant démarré normalement et se déclarait "healthy" (le healthcheck s'exécute dans le namespace réseau du conteneur, il ne détecte pas ce conflit côté hôte).
- Correction : le port hôte du service `redis` dans `docker-compose.yml` a été changé de `6379:6379` à `6380:6379` (le port interne au conteneur reste standard). `REDIS_URL` dans `.env` et `.env.example` pointe désormais vers `redis://localhost:6380/0`. Ce n'est pas un défaut de conception du projet mais une adaptation à un conflit d'environnement local ; le choix a été validé explicitement avant modification.
- Test de régression ajouté : aucun (problème d'infrastructure locale, pas de logique métier) — vérifié par observation directe des clés Redis (`docker compose exec redis redis-cli KEYS "cache:*"`) après appel à l'API, confirmant que le cache est bien écrit dans le conteneur du projet.
- Commit : à renseigner par l'utilisateur lors du commit correctif.
