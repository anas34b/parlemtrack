# BUG-001 — dépendance `requests` absente de requirements.txt

- Symptôme : lors de la génération initiale de `requirements.txt` via `pip3 freeze --user`, la dépendance `requests` (pourtant explicitement installée et requise par `pipeline/collect/downloader.py`) n'apparaissait pas dans le fichier produit, alors que toutes les autres dépendances demandées y figuraient.
- Cause racine : les dépendances Phase 0 avaient été installées avec `pip3 install --user --break-system-packages` au lieu d'un environnement virtuel isolé, faute de paquet système `python3-venv` disponible sans droits sudo au moment de l'installation initiale. Le paquet `requests` était déjà présent au niveau système (`/usr/lib/python3/dist-packages`, installé par `apt`) avec une version satisfaisant la contrainte demandée : `pip install --user` a donc considéré la dépendance comme déjà satisfaite et ne l'a pas installée dans le site `--user`, donc `pip freeze --user` ne l'a pas listée. Un `pip install -r requirements.txt` sur une machine sans ce paquet système aurait fait échouer le pipeline dès le premier `import requests`.
- Correction : passage à un environnement virtuel `.venv/` isolé (`python3 -m venv .venv`), qui ne voit aucun paquet système par défaut. Toutes les dépendances ont été réinstallées depuis zéro dans ce venv, et `requirements.txt` régénéré avec `pip freeze` depuis le venv activé — `requests==2.34.2` y figure désormais correctement, comme toutes les autres dépendances.
- Test de régression ajouté : aucun test automatisé dédié (problème d'environnement d'installation, pas de logique métier testable) — vérification manuelle systématique désormais intégrée à la Definition of Done de chaque phase : `pip install -r requirements.txt` dans un venv propre doit permettre d'importer tous les modules utilisés par le code.
- Commit : à renseigner par l'utilisateur lors du commit correctif.
