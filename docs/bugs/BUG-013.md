# BUG-013 — migrations Alembic non appliquées automatiquement au démarrage du backend en production

- Symptôme : lors du tout premier déploiement réel, le conteneur `backend` démarrait sans erreur mais l'API renvoyait des erreurs (schéma vide / tables manquantes) tant que `alembic upgrade head` n'était pas lancé à la main via `docker compose exec`.
- Cause racine : `backend/Dockerfile` définissait `CMD ["uvicorn", ...]` comme unique commande de démarrage — rien n'appliquait les migrations avant de servir des requêtes. Le test local de la Phase 5 avait lui-même nécessité un `alembic upgrade head` manuel après le `up --build`, sans que cette étape soit automatisée pour la suite.
- Correction : ajout de `backend/docker-entrypoint.sh` (`alembic upgrade head` puis `exec uvicorn ...`), utilisé comme `ENTRYPOINT` du conteneur `backend` à la place du `CMD` direct. L'étape `alembic upgrade head` manuelle, devenue redondante, a été retirée de `deploy.yml` et du manuel de déploiement.
- Test de régression ajouté : aucun test unitaire (défaut de configuration Docker, pas de code métier) — à vérifier lors du prochain déploiement réel (staging) que le conteneur `backend` applique bien les migrations seul, sans intervention manuelle.
- Commit : à renseigner par l'utilisateur lors du commit correctif.
