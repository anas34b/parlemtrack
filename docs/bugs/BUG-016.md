# BUG-016 — page liste des scrutins n'applique aucun filtre reçu par l'URL

- Symptôme : constaté en recette (cas F13, F14, 23/07/2026) — sur `/scrutins`, ni le filtre par groupe (`?groupe=<id>`) ni le filtre de période (`date_min`/`date_max`) n'ont d'effet sur la liste affichée : `?groupe=<id>` ne restreint rien (le test manuel initial semblait passer, mais était non concluant — voir plus bas), et un filtre de date est en plus perdu dès qu'on navigue vers la page suivante/précédente.
- Cause racine : confirmé dans `frontend/src/app/scrutins/page.tsx` — le composant ne lit que `page` depuis `searchParams` (`const { page } = await searchParams;`) et appelle `listerScrutins({ page, tailePage: TAILLE_PAGE })` sans jamais transmettre `groupe`, `date_min`, `date_max`, `type` ou `q`, alors que ces paramètres sont bien supportés côté API (`backend/app/api/routes/scrutins.py`) et déjà acceptés par `frontend/src/lib/api.ts::listerScrutins`. **Ce n'est pas seulement un problème de pagination** : la page liste n'applique aujourd'hui aucun filtre reçu par l'URL dès le premier chargement, avant même toute navigation entre pages. Les liens de pagination, cohérents avec ce constat plus large, ne reconstruisent eux aussi que `page` :
  ```tsx
  <a href={`/scrutins?page=${pageActuelle - 1}`}>← Plus récents</a>
  <a href={`/scrutins?page=${pageActuelle + 1}`}>Scrutins plus anciens →</a>
  ```
  Le test manuel initial du filtre par groupe (cas F13) avait été noté à tort comme réussi : la quasi-totalité des groupes politiques votent sur quasi tous les scrutins, donc constater que le groupe filtré apparaît dans le détail d'un scrutin de la liste ne prouve rien — ce serait vrai avec ou sans filtre actif. Seule la lecture du code a permis de confirmer que le paramètre n'était en réalité jamais transmis.
- Correction : **non corrigée à ce stade**. Deux changements nécessaires dans `frontend/src/app/scrutins/page.tsx` : (1) élargir le type de `searchParams` et transmettre `groupe`/`date_min`/`date_max`/`type`/`q` à `listerScrutins` dès le chargement initial de la page ; (2) reconstruire les liens de pagination à partir de l'ensemble des `searchParams` reçus, pas seulement `page`.
- Test de régression ajouté : aucun pour l'instant — sera ajouté avec la correction (vérifie qu'un `groupe`/`date_min`/`date_max` présent dans l'URL est bien transmis à `listerScrutins` dès le premier rendu, ET conservé dans les liens de pagination générés).
- Commit : à renseigner par l'utilisateur lors du commit correctif.
