# BUG-015 — recherche insensible à la casse mais pas aux accents

- Symptôme : constaté en recette (cas F08, 23/07/2026) — la recherche renvoie les mêmes résultats pour "sécurité sociale" et "SÉCURITÉ SOCIALE" (insensible à la casse), mais "SECURITE SOCIALE" (sans accents) ne renvoie aucun résultat alors que le même terme accentué en renvoie.
- Cause racine : `backend/app/services/recherche_service.py::rechercher` utilise `.ilike(f"%{q}%")` (SQLAlchemy → `ILIKE` PostgreSQL), qui applique une comparaison insensible à la casse selon la locale par défaut mais ne normalise pas les équivalences accentuées (`e` ≠ `é`) — ce n'est pas un comportement de `ILIKE`, ni du collate par défaut de PostgreSQL.
- Correction : **non corrigé à ce stade** — deux pistes identifiées, à trancher avant implémentation : (1) activer l'extension PostgreSQL `unaccent` et l'utiliser dans la requête (`unaccent(colonne) ILIKE unaccent(motif)`), ce qui nécessite une migration Alembic pour activer l'extension ; (2) normaliser le terme de recherche côté application (suppression des diacritiques via `unicodedata.normalize`) sur une colonne dédiée ou via une fonction SQL équivalente. Option (1) recommandée (traite aussi les données existantes sans réindexation applicative), à valider avant de coder.
- Test de régression ajouté : aucun pour l'instant — sera ajouté avec la correction (cas prévu : recherche avec/sans accents sur un terme connu, ex. "securite" vs "sécurité", doit renvoyer les mêmes résultats).
